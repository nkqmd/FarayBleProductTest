# Android 产测 App 约束规则

## 1. 目的

本文把需求、接口文档和服务端源码 `D:\test\py\test-server\app.py` 中已经确认的约束收敛成一份实现规则。

实现代码、AI 生成代码、联调脚本、测试用例都必须遵守本文。

## 2. 总原则

1. 服务端源码优先级高于接口文档示例。
2. 客户端必须做输入标准化，不能把脏数据直接交给服务端。
3. 所有“可重复执行”的动作都必须设计成幂等，尤其是结果上传。
4. 本地数据库是事实来源，服务端 `statistics` 只作为辅助视图，不作为当前会话唯一真相。
5. 扫描展示顺序必须稳定，只增不重排。
6. BLE 测试并发数以服务端下发的 `max_concurrent` 为准，客户端不额外引入手机能力探测、自动降级或人工覆盖并发上限。

## 3. 批次与配置规则

### 3.1 `batch_id`

1. `batch_id` 必须先 `trim()`。
2. `batch_id` 必须满足 `secure_filename(batch_id) == batch_id` 的约束。
3. 允许字符应按安全文件名理解，禁止空格、路径穿越、非预期特殊字符。
4. 客户端本地校验失败时不得发请求。

### 3.2 服务端批次目录约定

客户端默认服务端目录结构固定为：

```text
batch/
  <batch_id>/
    <batch_id>_mac_list.txt
    <batch_id>_config.json
    result/
```

客户端不得假设存在其他命名。

### 3.3 批次配置字段

以下字段视为服务端已确认必需字段：

- `expected_count`
- `expire_time`
- `expected_firmware`
- `device_type`
- `ble_name_prefix`
- `ble_config`

客户端规则：

1. 必须保留完整原始 `ble_config` JSON。
2. 必须做本地 schema 归一化。
3. 未识别字段不能丢弃，只能存入 `rawBleConfigJson` 供日志和后续兼容。

## 4. BLE 广播与 MAC 规则

### 4.1 广播名规则

1. 广播名必须以服务端下发的 `ble_name_prefix` 开头。
2. `ble_name_prefix` 不能为空、不能包含空白、必须以 `-` 结尾。
3. `ble_name_prefix` 之后必须紧跟 12 位十六进制 MAC。
4. 客户端不得写死 `_` 作为分隔符。
5. 如果产品协议需要 `_`，则 `_` 必须属于 `ble_name_prefix` 本身。

### 4.2 MAC 标准化规则

所有业务 MAC 在客户端内部统一标准化为：

- 展示格式：`12 位大写十六进制，不带分隔符`
- 存储格式：`Long`

示例：

- 输入 `c8:47:8c:c9:0f:e7`
- 标准化后展示为 `C8478CC90FE7`
- 存储为 48 位 `Long`

### 4.3 合法性判定顺序

必须严格按以下顺序处理扫描结果：

1. `RSSI` 过滤
2. 广播名 `prefix + 12HEX` 过滤
3. 解析业务 MAC
4. 白名单匹配
5. 历史已产测判定
6. 入测试队列

任一步失败都不得进入后续测试流程。

## 5. `ble_config` 归一化规则

### 5.1 当前已观测字段

服务端样例已出现字段：

- `scan_idle`
- `scan_active`
- `scan_duration`
- `rssi_min`
- `rssi_max`
- `max_concurrent`
- `retry_max`
- `retry_interval`
- `connect_timeout`
- `result_waiting_timed_out`
- `overall_testing_timed_out`
- `uuid_type`
- `service_uuid`
- `write_uuid`
- `notify_uuid`
- `service2_uuid`
- `write2_uuid`
- `notify2_uuid`
- `test_command`
- `expected_notify_value`
- `timeout_ms`

### 5.2 客户端归一化策略

0. `scan_idle` / `scan_active` 是 App 层扫描启停调度参数，不等同于 Android BLE 控制器层的 `scan interval/window`。
1. 优先读取 `scan_active`。
2. 若无 `scan_active`，且存在 `scan_duration` 且 `scan_duration > 0`，则把 `scan_duration` 映射为 `scanActiveMs`。
3. 若 `scan_duration <= 0` 或缺失，回退默认 `scanActiveMs = 50`。
4. `scanIdleMs` 默认 `100`。
5. `maxConcurrent` 至少为 `1`。
6. UUID 与 Hex 命令字符串必须先去空格再标准化大写。
7. 未通过校验的关键 BLE 字段不得直接用于测试。
8. `passDisplayMs`、`shutdownGraceMs` 在服务端未下发时由客户端本地默认配置提供。

### 5.2.1 UUID 归一化与第二组特征优先规则

当前实现已按以下规则落地，而且只改动了 UUID 配置归一化这一层：

1. 必须根据摘要里的 `uuid_type` 判断 UUID 类型。
2. `uuid_type = 16` 时，必须把短 UUID 转成标准 128 位 UUID。
3. `uuid_type = 32` 时，也必须兼容转换为标准 128 位 UUID。
4. 其他情况按 128 位 UUID 直接解析。
5. 当存在 `service2_uuid` 时，必须优先使用第二组 UUID：
   - `service2_uuid`
   - `notify2_uuid`
   - `write2_uuid`
6. 如果第二组里某个字段为空，则必须回退到第一组对应字段：
   - `service_uuid`
   - `notify_uuid`
   - `write_uuid`
7. 客户端不得把 `FFFF`、`FF02`、`FF01` 这类短 UUID 直接交给 `UUID.fromString(...)`。
8. 客户端从本地已保存批次恢复配置时，也必须重新按 `rawBleConfigJson` 做同样的 UUID 归一化，不能假设旧缓存已经是标准 128 位 UUID。

当前实现落点：

- 新增归一化器：`BleConfigNormalizer.kt`
- 摘要解析接入归一化器：`BatchApiClient.kt`
- 本地已保存批次从 `rawBleConfigJson` 恢复时也重新归一化：`BatchRepositoryImpl.kt`

当前实现已完成一次 `./gradlew.bat assembleDebug` 编译验证并通过。

联调时，`BleTestExecutor` 启动日志中的 UUID 应从原始短 UUID：

```text
service=FFFF notify=FF02 write=FF01
```

变为类似如下的标准 128 位 UUID：

```text
service=000000EE-0000-1000-8000-00805F9B34FB
notify=0000EE01-0000-1000-8000-00805F9B34FB
write=0000EE01-0000-1000-8000-00805F9B34FB
```

### 5.3 并发连接规则

1. 允许同时连接多个 BLE DUT。
2. 实际测试并发直接等于服务端下发的 `maxConcurrent`。
3. `maxConcurrent` 小于 `1` 时按 `1` 兜底。
4. 客户端只负责排队、去重和按并发槽位派发任务，不再引入额外并发概念。

## 6. 会话与 UI 规则

1. 每次点击“启动产测”必须创建一个新的 `session`。
2. 同一时刻只允许一个运行中的 `session`。
3. `session = RUNNING` 时必须锁定页面切换。
4. 只有 `session = STOPPED` 后才允许去结果页执行上传。
5. 列表项顺序由首次插入的 `sequence_no` 决定，后续状态更新不得改顺序。
6. 扫描到但超时未再次出现的设备只从当前产测页列表隐藏；本地记录不得删除。
7. 正在产测的 DUT 必须在 UI 上显示实时步骤状态，如 `CONNECTING`、`SUBSCRIBING`、`SENDING`、`WAITING_NOTIFY`、`DISCONNECTING`。
8. 测试结束后，`PASS` 或 `FAIL` 应持续显示在当前扫描列表上，直到设备超时不可见或转为 `ALREADY_TESTED`。
9. 通过后的 DUT 超过观察窗口仍被扫到时，必须转为 `ALREADY_TESTED`。

## 7. 白名单与本地存储规则

1. 白名单 MAC 不允许整体常驻内存 `HashSet<String>` 作为正式方案。
2. 白名单必须落 SQLite。
3. 表主键必须至少包含 `batch_id + mac_value`。
4. 白名单查询必须走索引。
5. 导入必须分块执行并支持失败恢复。
6. 历史是否“已产测”必须基于本地最终测试记录判断。
7. 当前会话中“刚通过又再次出现”的设备，也必须按“已产测再次出现”处理。
8. 当前会话中，设备首次发现、过程态更新、最终结果、非法判定都必须先写本地数据库，再驱动 UI 更新。
9. `PASS` / `FAIL` / `INVALID` 结果必须本地持久化，不能只存在内存，防止闪退、切后台或断电丢失。
10. “隐藏”只允许修改 `visible` 或等价展示字段，不允许删掉该设备对应的本地记录。

## 8. 认证规则

1. 除登录接口外，其余接口都必须带 `access_token`。
2. `/api/refresh` 必须使用 `refresh_token`。
3. refresh 成功后，旧 `access_token` 和旧 `refresh_token` 都立即作废。
4. 客户端必须只保留最新一套 token。
5. token 刷新必须串行化，防止并发刷新互相覆盖。
6. 服务端 token 状态只保存在内存中，服务重启后会丢失，客户端不得依赖服务端记忆。

## 9. 结果上传规则

### 9.1 上传请求必填字段

客户端必须发送以下字段：

- `report_id`
- `report_digest`
- `batch_id`
- `factory_id`
- `app_version`
- `test_start_time`
- `test_end_time`
- `statistics`
- `success_records`
- `fail_records`
- `invalid`

### 9.2 `report_id` 规则

1. `report_id` 必须先 `trim()`。
2. `report_id` 必须满足安全文件名约束。
3. `report_id` 必须以 `<batch_id>_` 开头。
4. 推荐格式：`<batch_id>_<report_digest前16位>`。

原因：

- 服务端落盘文件名来自 `report_id`
- 服务端 `statistics` 只检索 `<batch_id>_*.json`

不满足第 3 条时，上传可能成功，但统计接口可能看不到这条结果。

### 9.3 `report_digest` 规则

1. 必须使用 SHA-256。
2. 必须输出 64 位小写十六进制字符串。
3. 必须基于“客户端请求体摘要源对象”计算。
4. 不得包含服务端追加字段：
   `upload_id`、`uploaded_by`、`uploaded_at`、`batch_mac_count`

### 9.4 摘要源对象的稳定化规则

为了确保重复上传命中服务端幂等，`report_digest` 的输入必须稳定：

1. JSON key 顺序固定。
2. 所有时间统一 ISO 8601。
3. 所有 MAC 统一为 12 位大写十六进制。
4. `success_records` 按 `mac ASC, time ASC` 排序。
5. `fail_records` 按 `mac ASC, time ASC, reason ASC` 排序。
6. `invalid` 按 `mac ASC, time ASC` 排序。

### 9.5 幂等规则

服务端去重键为：

```text
batch_id | factory_id | report_digest
```

客户端必须遵守：

1. 同一会话重试上传时必须复用同一个 `report_id` 和 `report_digest`。
2. 只要请求内容改变，就必须重新计算 `report_digest`。
3. 收到服务端 `duplicate = true` 也视为成功。

## 10. 结果统计规则

1. 本地结果页主统计必须来自本地数据库聚合。
2. 服务端 `/api/production/statistics` 只能作为“服务端最近一份已接收记录”的参考值。
3. 补传旧会话可能改变服务端“最新结果”，所以不能用它覆盖当前本地会话统计。
4. 结果统计必须通过 SQL 聚合当前 session 数据完成，不能把几万到 10 万条记录全量读入内存后再统计。
5. 当前产测页设备列表必须只查询当前 session 且 `visible = 1` 的记录，并使用索引与分页/增量刷新策略。

## 11. 通过后停播规则

### 11.1 业务含义

“通过”不只是测试成功，还隐含一个工艺要求：

1. DUT 通过后应在约定时间内停止广播或关机。
2. 如果通过后一直广播，App 无法可靠区分它是“当前 DUT 未关机”还是“新出现的重号 DUT”。
3. 因此“通过后停播”本身就是产测判定链的一部分。

### 11.2 UI 状态规则

推荐状态语义：

1. `CONNECTING` / `SUBSCRIBING` / `SENDING` / `WAITING_NOTIFY` / `DISCONNECTING`: 当前执行到的产测步骤。
2. `PASS`: 当前设备本次测试通过，并持续显示为“通过”。
3. `FAIL`: 当前设备本次测试失败，并持续显示为“失败”。
4. `ALREADY_TESTED`: 已产测再次出现。

状态迁移规则：

1. 测试任务启动后，UI 按实际执行步骤更新过程态。
2. 测试成功立即进入 `PASS`，测试失败立即进入 `FAIL`。
3. `PASS` / `FAIL` 不再自动切换到额外中间态。
4. 若在 `shutdownGraceMs` 内设备停播，则设备从当前扫描列表消失。
5. 若超时后仍被再次扫描到，则更新为 `ALREADY_TESTED`。

### 11.3 “已产测”只有两类合理解释

对一个已通过的业务 MAC 再次出现，从 App 视角通常只能推断为两类情况之一：

1. 刚通过的 DUT 一直未关机。
2. 现场出现了另一台使用相同名称规则的 DUT。

如果广播名和地址也相同，App 一般仍无法可靠区分这两类情况，所以：

1. UI 可以统一显示 `已产测`。
2. 内部 reason 必须记录为 `NOT_POWERED_OFF_OR_DUPLICATE_DEVICE`。
3. 该设备不得重新进入测试流程。

## 12. 客户端发送前自校验规则

发送上传请求前，客户端必须至少校验：

1. `batch_id` 非空且安全
2. `factory_id` 非空且已 `trim()`
3. `report_id` 非空、安全、且以 `<batch_id>_` 开头
4. `report_digest` 为 64 位小写 hex
5. `test_end_time >= test_start_time`
6. `statistics` 五个核心字段完整
7. `success_records` 每项包含 `mac`、`time`
8. `fail_records` 每项包含 `mac`、`result`、`reason`、`time`
9. `invalid` 每项包含 `mac`、`time`

## 13. 不允许的实现

以下做法禁止进入正式实现：

1. 把整个白名单读入 `HashSet<String>` 作为主判定结构。
2. 直接把 `BluetoothDevice.address` 当成业务 MAC。
3. 写死广播名规则为“固定下划线分隔”。
4. 每次重试上传都重新随机生成 `report_id`。
5. 依赖服务端 `statistics` 作为当前本地会话唯一真相。
6. 在产测运行中允许用户切走页面继续操作其他页。
7. 设备测试完成后不落本地库，只依赖当前内存状态驱动 UI。
8. 把“通过后再次出现”的设备重新投入测试。
9. 脱离服务端配置，擅自引入额外并发上限、自动降级或固定单连接策略。

## 14. 推荐默认值

在服务端字段缺失时，客户端使用以下默认值：

- `scanIdleMs = 100`
- `scanActiveMs = 50`
- `maxConcurrent = 1`
- `connectTimeoutMs = 10000`
- `notifyTimeoutMs = 30000`
- `overallTimeoutMs = 60000`
- `passDisplayMs = 3000`
- `shutdownGraceMs = 3000`

这些默认值仅用于兜底，真实运行仍以服务端配置为主。
